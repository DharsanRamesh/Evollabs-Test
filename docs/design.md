Architecture overview
----------------------

- **Teleport core** runs as a single container (`teleport` service in `docker-compose.yml`) using the official Teleport distroless image.
- Teleport is configured (via `teleport/teleport.yaml`) to run:
  - `auth_service` – local user database and certificate authority.
  - `proxy_service` – public entrypoint for `tsh` and the Web UI.
  - `ssh_service` – acts as the SSH node that users connect to via Teleport.
  - `app_service` – publishes an internal HTTP app (`demo-app`) for Application Access.
- A separate `app` container (`targets/app`) runs a minimal Python HTTP server that:
  - Exposes `/healthz` and `/`.
  - Echoes Teleport‑injected headers so you can see `user` and `roles` values.

Key security decisions and trade‑offs
-------------------------------------

- **Local‑only, lab‑grade configuration**:
  - The `auth_token` in `teleport/teleport.yaml` is a static value used only within this local Docker network.
  - All Teleport state (CA keys, certs, tokens) is persisted under `teleport/data` and never committed to git.
- **Roles are explicit and minimal**:
  - `admin` can manage everything.
  - `engineer` can:
    - SSH into nodes labeled `env=lab, service=ssh-node`.
    - Access applications labeled `env=lab, service=demo-app`.
  - `readonly` can:
    - Access the `demo-app` application.
    - **Cannot** obtain an SSH shell (denied logins).
- **Non‑interactive identities for tests**:
  - Instead of interactive `tsh login`, tests use identity files generated by `tctl auth sign`.
  - This keeps the smoke tests fully automated and repeatable.

What would change for a hardened production design
--------------------------------------------------

In a real production deployment you would:

- Run Teleport Auth and Proxy on separate instances, with HA and dedicated storage.
- Use short‑lived join tokens or Auto‑discovery instead of a static `auth_token`.
- Integrate Teleport with an external identity provider (e.g. SSO / OIDC) instead of local users.
- Lock down roles further:
  - Restrict SSH capabilities with enhanced `node_labels`, `kubernetes_labels`, or per‑command restrictions.
  - Use `deny` blocks more aggressively and require reasoned access workflows (e.g. access requests).
- Terminate TLS with proper certificates (DNS records + public CA).
- Push logs and audit events to a central system (e.g. SIEM) and set up alerting on denied events and anomalies.
